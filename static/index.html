🔥 Flamegraph
How to read
  • width Width = time spent. Wider bar = more CPU time consumed by that function and everything it calls.
  • depth Each row is one level deeper in the call stack. The root bar is at the top; leaf functions are at the bottom.
  • color Colors are decorative — assigned by function name, not meaning. They help distinguish adjacent frames, nothing more.
  • click Click any frame to zoom in and focus on that subtree. Use Zoom out or Reset zoom to step back out.
  • search Type in the search box to highlight matching frames across the entire graph.
What to look for
  • Wide bars near the top — expensive code paths. The wider, the more time your program spends going through that function.
  • Flat plateau at the bottom — a leaf function doing heavy self-work (no further calls). Common culprits: tight loops, string ops, regex, JSON parsing.
  • Many thin slivers — time is spread across lots of small functions. Usually fine, but can indicate excessive abstraction or serialization overhead.
  • One tower dominating — a single code path consumes most CPU. Good starting point for optimization.
  • Unexpected frames — GC, serialization, or logging functions appearing wide often reveal hidden costs.
Hover tooltip
  • % — percentage of all samples that passed through this frame (inclusive of children).
  • self % — time spent in this function alone, excluding callees. High self % = this function itself is the bottleneck.
  • Frames narrower than ~2 px are still clickable to zoom — use zoom to explore crowded areas.
